Q1: FDB decouples these 2 components to allow for each of them to be scaled easily and improve failure handling. The TMS handles assigning increasing read and commmit versions, providing MVCC read versions, dealing with transaction commits, and implementing lock free concurrency control to detect read-write conflicts. The SS stores data, gives clients the read response directly, pulls mutation logs, and is a storage engine.
Q2: The resolver is responsible for detecting read-write conflicts. This check happens when client attempts to commit data and Resolver checks for conflicts before deciding to abort the transaction or allow it to happen.
Q3: The read version is a number that represents a consistent snapshot of the database that the data the user was given belonged to. This helps with ensuring the strict serializability gurantee FDB provides.
Q4: Snapshot reads don't add to the read conflict range so they never abort. Snapshot reads are ideal for tasks that don't require strict serializabilty or can be performed in the background and have both a read and write part to them. Normal reads do add to the read conflict ranges and thus can conrtibute to a transaction being aborted. Normal reads should be used when strict serializability is required.

Thus the only information that was added was the data in the old logServers from the [PEV,RV] which makes this more of a metadata update and less of a data update which could likely take much longer. The data is the updated by the storage servers in the background as they normally do with asyncrhonous pulls to find any mutations from the logServers.
